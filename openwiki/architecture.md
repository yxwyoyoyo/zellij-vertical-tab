# Plugin architecture and domain model

The plugin is a single Rust binary compiled to `wasm32-wasip1`. Zellij loads it, calls the lifecycle generated by `register_plugin!(State)`, supplies tab and mouse events, and hosts the rendered `Text` output. The [quickstart](quickstart.md) maps the surrounding repository; the [development guide](development.md) explains how this architecture is built and verified.

## System boundary

The plugin owns only presentation state and interaction mapping. Zellij remains authoritative for tabs, active status, pane dimensions, theme rendering, permissions, and actual tab switching.

```text
Zellij tab state ──TabUpdate──> State.tabs / active_idx
                                    │
                                    ├── visible_window ──> scroll_offset
                                    │
Pane rows + columns ──render────────┴──> formatted Text rows ──> Zellij host

Zellij mouse event ──> row + scroll_offset ──> switch_tab_to(position + 1)
```

This boundary keeps the internal algorithms pure enough for host-target unit tests while delegating terminal rendering and application mutation to `zellij-tile` (`src/main.rs`).

## Packaging and lifecycle

`Cargo.toml` declares an explicit `[[bin]]` target named `zellij_vertical_tab`. This is not incidental: Zellij's loader expects a command-style WASM module exporting `_start`, and `register_plugin!` generates the `main()` entrypoint needed by that form. The checked-in lessons report that a cdylib/reactor attempt failed to load because that export was absent.

### `load()`

`State::load` requests:

- `ReadApplicationState`, required to receive tab information.
- `ChangeApplicationState`, required to switch tabs.

It then subscribes to `TabUpdate` and `Mouse`. Configuration is currently ignored, so pane width and placement are layout concerns rather than plugin options.

### `update()`

The first received event triggers `set_selectable(false)` exactly once through `unselectable_set`. Deferring this call is a runtime safety measure, not a style preference; invoking it during `load()` was observed to kill a Zellij 0.44 client in the default-tab-template layout.

After that setup, `update()` dispatches only the subscribed event variants:

- `TabUpdate` refreshes tab state and requests rendering only when the supplied tab vector differs.
- `Mouse::LeftClick` maps a content row to a tab and asks Zellij to switch it; it waits for the resulting `TabUpdate` to request a rerender.
- `Mouse::ScrollUp` and `ScrollDown` mutate the offset by one row and request rendering only if the offset changed.
- Other mouse and event variants are ignored.

### `render()`

`render(rows, cols)` first stores a changed pane height and recomputes the visible window. Empty tabs or zero dimensions produce no output. Otherwise it renders at most one row per pane row using `print_text_with_coordinates` at column `0`, applying `Text::selected()` to the active tab.

Zellij interprets the emitted `ztext` APC sequences with the user's theme. The plugin therefore does not subscribe to mode/theme updates or calculate colors itself.

## State model

`State` has five fields:

| Field | Meaning | Updated by |
| --- | --- | --- |
| `tabs: Vec<TabInfo>` | Latest ordered tab snapshot from Zellij | `TabUpdate` |
| `active_idx: Option<usize>` | Active tab's vector index, if present | `TabUpdate` |
| `scroll_offset: usize` | Vector index shown on the first row | Tab changes, scrolling, and resize |
| `rows: usize` | Last rendered content height | `render()` |
| `unselectable_set: bool` | Guard ensuring deferred selectability setup runs once | First `update()` |

`active_idx` and `scroll_offset` are vector indices, whereas `TabInfo.position` is converted to a one-based number for display and `switch_tab_to`. Maintaining that distinction is essential when adding filtering, sorting, or synthetic rows.

## Event and interaction workflows

### Tab synchronization and active-tab following

1. Zellij sends `TabUpdate(Vec<TabInfo>)`.
2. Identical snapshots return `false` and avoid a redundant render.
3. The plugin locates the active vector index and replaces `tabs`.
4. If active status moved, `visible_window` minimally shifts the window so the active tab is visible.
5. If active status did not move, `clamp_offset` only repairs an invalid offset, such as after tabs shrink.
6. Returning `true` asks Zellij to render.

This is why keyboard-driven tab changes remain visible even though the plugin handles no keyboard events.

### Click-to-switch

A nonnegative click row is added to `scroll_offset`. If the resulting vector index exists, the plugin calls `switch_tab_to(tab.position + 1)`. Out-of-range and negative rows are safe no-ops. The pane can receive these mouse events even after becoming unselectable, an empirically verified behavior also used by Zellij's built-in tab bar.

### Wheel scrolling

Scroll-up uses saturating subtraction. Scroll-down increments then applies `clamp_offset`, so the final page cannot move beyond the last complete visible window. Both paths compare old and new offsets to avoid unnecessary renders at the boundaries.

## Domain rules

### Visible-window calculation

`clamp_offset(tab_count, offset, rows)` returns:

- `0` when the pane has no rows or all tabs fit.
- Otherwise, the lesser of the requested offset and `tab_count - rows`.

`visible_window(tab_count, active, previous, rows)` starts from that clamped offset. If an active tab exists and the pane has rows, it moves the offset only when the active index lies above the first row or at/after the exclusive lower boundary. This “minimal movement” rule preserves manual scroll position whenever the active tab is already visible.

### Row format

Each row is conceptually:

```text
<lead><right-aligned index> <name>
```

The lead is `▲` on the first visible row when tabs are hidden above, `▼` on the last visible row when tabs are hidden below, and a space otherwise. Index width is based on the total tab count, keeping single-digit indices aligned when the session reaches ten or more tabs.

`fit_to_width` truncates by Rust `char` count and pads short text to the full column width. Padding allows selected styling to span the row. This is not a full terminal-cell-width implementation: comments rely on Zellij to clip display overflow for wide characters. Changes involving CJK text, combining characters, or emoji should add explicit runtime coverage.

## Runtime and layout constraints

The supplied `zellij.kdl` places the plugin in a borderless 20-column pane inside `default_tab_template`, with `children` in a separate sibling pane and the built-in status bar below. As a result, every new tab inherits the vertical list and the built-in horizontal tab bar is absent.

Two constraints jointly stabilize this arrangement:

- The plugin becomes unselectable only after startup reaches its first event.
- `children` remains wrapped in `pane { ... }`; it must not be a direct sibling of the unselectable plugin within the split.

The task notes attribute the crash to Zellij-side layout geometry and report controlled PTY experiments rather than a proven upstream root-cause fix. Treat layout refactoring as a runtime-sensitive change and follow the [headless verification workflow](development.md#headless-runtime-verification).

## Integration points

- **Zellij plugin ABI and events:** `zellij-tile = 0.44.3` supplies lifecycle macros, permissions, `TabInfo`, events, rendering, selectability, and tab-switch commands.
- **WASI build target:** `wasm32-wasip1` produces the module Zellij loads.
- **Zellij layout KDL:** `zellij.kdl` determines plugin location, width, border, inheritance, and surrounding panes.
- **User theme:** host rendering resolves `Text::selected()`; no plugin-side color configuration exists.
- **Permission cache:** Zellij caches grants by plugin location, according to `tasks/lessons.md`; changing the path can cause a new prompt.

Build compatibility and deployment consequences of these integrations are covered in [development and operations](development.md#compatibility-and-release-notes).

## Safe extension points

- New pure formatting or viewport behavior belongs beside `format_row`, `fit_to_width`, `clamp_offset`, and `visible_window`, with colocated unit tests.
- New mouse interactions belong in the `Event::Mouse` match, but synthetic rows such as “new tab” require separating visual-row mapping from direct `tabs[offset + line]` indexing.
- New Zellij event-driven features require adding subscriptions in `load()` and handling their permission implications.
- User configuration can use the currently ignored `BTreeMap<String, String>`, but width is still controlled by KDL unless plugin and layout conventions are coordinated.
- Layout changes must preserve or deliberately revalidate startup/selectability invariants.

After any change, use the checks in the [development guide](development.md#testing-strategy), especially a real Zellij launch for lifecycle, permission, mouse, or layout modifications.
